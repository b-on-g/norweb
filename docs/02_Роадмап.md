# RAGU Live Demo — Роадмап

> Документ 3 из 3 (п.5). Поэтапный план разработки. **Принцип: 1 этап = 1 Merge Request** в репозиторий RAGU на GitHub, по возможности с тестами в том же MR; CI/CD накатываем после стабилизации каркаса.
> Связанные: `docs/01_Концепция_функциональность_стек.md`, `wireframe/RAGU Wireframe.dc.html`.

---

## 0. Принципы работы

- **1 этап = 1 MR.** Маленький, проверяемый, с описанием и чек-листом Acceptance Criteria.
- **Тесты в том же MR.** Бэкенд — pytest + детерминированный mock-LLM; фронт — smoke/e2e на ключевых сценариях.
- **Mock-first.** Контракт API и mock-данные — фреймворк-агностичны (JSON). Сначала фронт работает на моках; дальнейшая замена источника данных возможна только как backend adapter без live-indexing API.
- **Definition of Done этапа:** код + тесты зелёные в CI + обновлён mock/контракт + краткое демо (gif/скрин) в описании MR + ветка ревьюится и мёржится.

## 1. Временные рамки

| Веха | Срок | Что должно быть |
|---|---|---|
| Каркас + решения | **конец недели 1** (≈27–29 июня) | Кликабельный каркас фронта; **утверждённый список предындексированных датасетов** (нужно согласовать с руководителем); API capabilities фиксируют отсутствие upload/live-indexing |
| MVP v1 | **начало недели 2** (≈30 июня) | Граф + чат + дашборд на предындексах (реальные данные, mock-инференс) |
| MVP финал | **мягкий дедлайн 1 июля / жёсткий 5 июля** | Все разделы + настройки + экспорт на предындексированных датасетах; публичный URL |
| Подача EMNLP | **10 июля (AoE)** | Буфер: видео-скринкаст по готовому стенду + финальные правки |

> Реальный дедлайн трека — 10 июля, это даёт ~5 дней буфера после внутреннего жёсткого срока 5 июля на видео и полировку.

## 2. Зависимости и параллельные треки

- **Трек согласований (вне кода, критичен для сроков):** (а) утверждение датасетов с руководителем; (б) подготовка/пересчёт предындексов выбранных корпусов. GPU/Yandex Cloud/live-indexing больше не являются зависимостями web demo.
- **Технический критический путь:** Контракт API (Этап 0) → предындексы (Этап 2) → граф (Этап 3) → чат (Этап 4).

## 3. Этапы (MR-ы)

### Этап 0 — Каркас фронта + контракт API `[MR #1]`
**Цель:** скелет приложения и зафиксированный фреймворк-агностичный API-контракт + mock-данные.
**Объём:**
- App shell: левый навигатор, верхняя панель (датасет, пресет, настройки, экспорт, EN/RU), роутинг между разделами.
- Пустые экраны-заглушки всех 5 разделов по wireframe.
- JSON-схемы ответов (датасеты, граф node-link, сообщения чата + трейс, метрики дашборда, конфиг движка) + фикстуры-моки.
- i18n-каркас EN/RU.

**Acceptance Criteria:**
- [ ] Переходы между всеми разделами работают; активный датасет и пресет видны глобально.
- [ ] Все ответы рисуются из mock-фикстур, соответствующих опубликованным схемам.
- [ ] Переключатель EN/RU меняет строки интерфейса.
- [ ] Layout соответствует утверждённому wireframe.

**Тесты:** snapshot/контрактные тесты схем; smoke-навигация (e2e).
**Риски:** не зафиксирован фронт-стек ($mol vs React) → *митигация:* контракт и моки агностичны, выбор стека не блокирует мёрж этапа.

---

### Этап 1 — Раздел «Датасеты» (Gallery) `[MR #2]`
**Цель:** витрина корпусов и точка входа в контекст.
**Объём:** карточки датасетов (метаданные, мини-превью, бейджи узлы/связи/сообщества); выбор корпуса → глобальный контекст; кнопки «Загрузить документ» / «Загрузить готовый индекс» скрываются или блокируются по `GET /api/v1/capabilities`.
**Acceptance Criteria:**
- [ ] ≥3 датасета из утверждённого списка отображаются с корректными метаданными.
- [ ] Выбор датасета меняет контекст для разделов Граф/Чат/Дашборд.
- [ ] Кнопки загрузки не запускают indexing flow; UI читает `upload_document=false`, `upload_index=false`, `live_indexing=false`.

**Тесты:** рендер галереи из фикстур; смена контекста.
**Риски:** датасеты не утверждены/не посчитаны → *митигация:* использовать 1–2 готовых корпуса (медицина из бенчей, наша статья) как заведомо доступные.

---

### Этап 2 — Предындексы на бэкенде `[MR #3]`
**Цель:** реальные данные графа из посчитанных индексов RAGU.
**Объём:** FastAPI-эндпойнты, грузящие предындексы (NetworkX GML + векторное/KV хранилище) в память; отдача графа (node-link + закешированный ForceAtlas2-layout), метаданных, статистики; Pydantic-схемы из доменных типов RAGU.
**Acceptance Criteria:**
- [ ] `GET /datasets`, `GET /datasets/{id}/graph`, `GET /datasets/{id}/stats` отдают реальные данные ≥2 корпусов.
- [ ] Граф приходит с предрасчитанными координатами (браузер не пересчитывает layout).
- [ ] Ответы соответствуют схемам Этапа 0 (контрактные тесты зелёные).

**Тесты:** pytest на эндпойнты с фикстурами-индексами; контракт ↔ фронт.
**Риски:** большие графы тяжело отдавать целиком → *митигация:* серверный семплинг/агрегация, пагинация окрестностей, лимит видимых узлов.

---

### Этап 3 — Граф (Explorer) `[MR #4]`
**Цель:** интерактивный граф знаний.
**Объём:** Sigma.js + graphology, цвет = тип NEREL, толщина = strength; zoom/pan/drag; карточка сущности (provenance-чанки); фильтры (тип, поиск узла, порог strength); слой сообществ Leiden + их отчёты; «развернуть окрестность»; лимит узлов для производительности.
**Acceptance Criteria:**
- [ ] Граф ≥1000 узлов рендерится и интерактивен без фризов.
- [ ] Клик по узлу → карточка с типом, описанием, связями, исходными чанками.
- [ ] Фильтры по типу и порогу strength работают; поиск узла фокусирует его.
- [ ] Переключатель сообществ подкрашивает кластеры и открывает отчёт.

**Тесты:** e2e на клик/фильтр/поиск; unit на трансформацию graph→sigma.
**Риски:** «шар» на больших корпусах (см. скриншот) → *митигация:* прогрессивная подгрузка от выбранного узла, серверная агрегация, дефолтный лимит.

---

### Этап 4 — Чат + трейс `[MR #5]`
**Цель:** диалог с агентом по графу с объяснимостью.
**Объём:** чат-UI со стримингом; селектор поискового движка (Local/Global/Naive/Mix/QueryPlan); панель трейса (сущности/чанки/сообщества в контексте + подсветка на графе + движок/top-k/rerank + тайминги/энергия запроса); затравки-вопросы по датасету. Бэкенд: эндпойнт запроса (mock-инференс → реальный поиск RAGU поверх предындекса).
**Acceptance Criteria:**
- [ ] Вопрос возвращает стримящийся ответ + структурированный трейс.
- [ ] Узлы из трейса подсвечиваются в Explorer.
- [ ] Смена движка меняет поведение/трейс.
- [ ] Затравки подставляются в инпут.

**Тесты:** контракт запроса/трейса; e2e «задать вопрос → увидеть подсветку».
**Риски:** инференс ещё mock → *митигация:* реальный **поиск** по предындексу можно включить раньше инференса; генерацию домокать.

---

### Этап 5 — Настройки движка (пресеты + ручной режим) `[MR #6]`
**Цель:** управление конфигурацией RAGU.
**Объём:** slide-over панель; пресеты ⚡Быстро/🎯Точно/🎬Демо в шапке; ручные группы по 6 шагам пайплайна (chunking, extraction+ICL+валидация+модель, summarization, communities, refinement, search+rerank+hybrid+top-k); бейдж «требует переиндексации» на настройках индексации; настройки поиска применяются к запросу сразу.
**Acceptance Criteria:**
- [ ] Выбор пресета выставляет полный набор полей ручного режима.
- [ ] Настройки поиска (шаг 6) меняют ответ/трейс без переиндексации.
- [ ] Настройки индексации помечены «требует переиндексации» и уходят в флоу индексации.
- [ ] Конфиг сериализуется в формат, принимаемый бэкендом RAGU.

**Тесты:** unit на маппинг пресет→конфиг; контракт конфига.
**Риски:** перегруз UI → *митигация:* по умолчанию свёрнут, ключевое — в шапке.

---

### Этап 6 — Дашборд `[MR #7]`
**Цель:** аналитика и бизнес-метрики.
**Объём:** статистика графа (узлы/связи/сообщества, распределение типов, топ-сущности); метрики качества из бенчмарков (AC, Evidence Recall, CR, ROUGE-L; RAGU vs LightRAG/HippoRAG2/Naive); стоимость/энергия (кВт·ч = `TDP×время×PUE`, $ и экономия vs GPT-4-class API); тайминги пайплайна по шагам; таблица логов запросов с разворачиваемым трейсом. Чарты — изолированный «островок» (та же стратегия, что для графа).
**Acceptance Criteria:**
- [ ] Все 4 блока заполнены реальными данными корпуса + статичными бенч-метриками из статьи.
- [ ] Энергокалькулятор считает по формуле и показывает $-экономию vs API.
- [ ] Логи запросов разворачиваются в трейс.

**Тесты:** unit на формулу энергии/стоимости; рендер из фикстур.
**Риски:** реальных таймингов нет (mock-инференс) → *митигация:* показывать оценки с явной пометкой «estimated», заменить на реальные после подключения backend-метрик.

---

### Этап 7 — Экспорт и логи `[MR #8]`
**Цель:** выгрузка артефактов.
**Объём:** граф (GraphML/GEXF/JSON/PNG); отчёты сообществ (MD/JSON); чат+трейсы (MD/JSON); метрики (CSV/JSON); логи сервиса (JSONL).
**Acceptance Criteria:**
- [ ] Каждый раздел имеет рабочую кнопку экспорта в заявленных форматах.
- [ ] Экспортируемые файлы валидны и открываются (GraphML — в Gephi/Cytoscape).

**Тесты:** валидация форматов экспорта.
**Риски:** низкие.

---

### Этап 8 — Upload/live-indexing исключены `[removed]`
**Цель:** явно зафиксировать отсутствие upload/job/live-indexing API, чтобы фронт не строил интеграцию на несуществующих ручках.
**Объём:** `GET /api/v1/capabilities` возвращает `upload_document=false`, `upload_index=false`, `live_indexing=false`, `job_queue=false`, `gpu_worker=false`; upload/job endpoints отсутствуют в OpenAPI.
**Acceptance Criteria:**
- [ ] Swagger/OpenAPI не содержит upload/job/live-indexing endpoints.
- [ ] Фронт скрывает или блокирует загрузку документа/индекса по capabilities.

**Тесты:** контрактный тест `/openapi.json`; тест `/api/v1/capabilities`.
**Риски:** низкие; решение осознанно сужает scope.

---

### Этап 9 — Деплой + публичный URL `[MR #10]`
**Цель:** ссылка для ревьюеров и съёмки видео.
**Объём:** фронт + FastAPI backend; публичный туннель (Cloudflare Tunnel) или другой простой хостинг без Redis/GPU/Yandex-зависимостей.
**Acceptance Criteria:**
- [ ] Демо доступно по публичному URL извне.
- [ ] Холодный старт воспроизводим из `docker compose up`.
- [ ] README с инструкцией запуска (требование трека — installable package как альтернатива URL).

**Тесты:** smoke по публичному URL.
**Риски:** стабильность туннеля/квоты облака → *митигация:* installable-package как запасной вариант (разрешён правилами трека).

---

### Этап 10 — CI/CD `[MR #11]`
**Цель:** автоматизация качества.
**Объём:** GitHub Actions — lint + pytest + e2e на каждый PR (детерминированные mock-данные, без ключей/GPU → секунды); CD-деплой по тегу.
**Acceptance Criteria:**
- [ ] PR не мёржится с красным CI.
- [ ] Тег → автодеплой демо.

**Риски:** низкие; накатываем после стабилизации каркаса (как и договорено).
