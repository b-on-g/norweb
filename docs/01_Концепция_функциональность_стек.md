## 0. Контекст и зачем это нужно

Мы подаём RAGU на **EMNLP 2026 System Demonstration Track**. По требованиям трека (single-blind, можно раскрывать авторство и бренд) для подачи **обязательны три артефакта**, иначе desk reject:

| Артефакт | Требование трека | Наша ситуация |
|---|---|---|
| Статья | ≤ 6 страниц + appendix ≤ 2 стр., формат EMNLP, **обязательна секция evaluation** | Готова (`EMNLP26_Industry_RAGU`) |
| Скринкаст-видео | ≤ 2.5 мин, screencast с озвучкой, можно на YouTube | Снимаем **на основе этого live demo** |
| **Live demo website / installable package** | **строгое требование — ссылка обязательна**, иначе desk reject | **← это наш проект** |

**Ключевые даты:** дедлайн подачи — **10 июля 2026** (AoE), нотификация — 20 августа, camera-ready — 30 августа, конференция — 24–29 октября, Будапешт.

**Двойная цель проекта:**
1. **Минимум (к подаче):** работающий публичный стенд, по которому снимаем видео и который даём ссылкой ревьюерам.
2. **Максимум (на будущее):** постоянная витрина технологии — клиент заходит, за минуту оценивает силу RAGU без развёртывания, индексации, подбора моделей и поиска мощностей.

**Что демонстрируем (тезис статьи →  что должен «пощупать» пользователь):**
- Компактная доменно-адаптированная модель **meno-lite 7B** делает построение графа знаний не хуже моделей в ~4× больше → *«смотри, какой чистый граф собран локальной 7B-моделью»*.
- Многошаговый пайплайн (extraction → консолидация) даёт связнее граф и выше Answer Correctness, чем single-pass LightRAG → *«покрути тумблеры пайплайна и сравни»*.
- Инженерная зрелость и дешёвый локальный инференс → *«вот сколько кВт·ч и $ ты экономишь против GPT-4-class API»*.

## Глава 1. Функциональность демо (что основное должно быть)

### 1.1 Карта продукта (app shell)

Одностраничное приложение с левым навигатором и общим контекстом «активный датасет + активный пресет движка». Пять основных разделов:

| # | Раздел | Суть | Статус в MVP |
|---|---|---|---|
| 1 | **Датасеты** (Gallery) | Витрина предындексированных корпусов + загрузка своего | ✅ ядро MVP |
| 2 | **Граф** (Explorer) | Интерактивный граф знаний (Obsidian-like) + карточка сущности | ✅ ядро MVP |
| 3 | **Чат** (Agent) | Диалог с агентом по конкретному графу + трейс ответа | ✅ ядро MVP |
| 4 | **Дашборд** | Статистика графа, метрики качества, энергия/стоимость, тайминги, логи | ✅ ядро MVP |
| 5 | **Настройки движка** | Пресеты + ручной режим (slide-over) | ✅ ядро MVP |
| — | Индексация своего документа / готового индекса | Загрузка → прогресс по шагам пайплайна | 🟡 mock в начале, live к концу MVP |
| — | Экспорт и логи | Выгрузка графа, отчётов сообществ, чата, логов сервиса | ✅ ядро MVP |
| — | Переключатель языка интерфейса | EN / RU | ✅ ядро MVP |

### 1.2 Раздел «Датасеты» (точка входа)

**Цель:** за 10 секунд дать пользователю интересный граф, который можно сразу крутить и спрашивать.

- Галерея карточек предындексированных корпусов. Для каждого: название, домен, иконка, краткое описание, мини-превью графа, бейджи (узлы / связи / сообщества, на какой модели и пресете построен).
- **Предлагаемый набор корпусов** (интересны сами по себе + покрывают разные домены и оба языка):
  1. **Юриспруденция РФ** — подборка кодексов / ФЗ (право — наглядная сеть норм, отсылок, органов).
  2. **ООН** — Устав ООН + ключевые резолюции (международные органы, страны, статьи).
  3. **Научные статьи** — наша статья + цитируемые работы (авторы, методы, бенчмарки, цитирования — мета-демонстрация «граф про сам GraphRAG»).
  4. **Медицина** — GraphRAG-Bench (Medical), уже есть в бенчах → можно показывать рядом метрики качества из статьи.
  5. **Википедия-подборка по теме** — связный тематический срез (демонстрирует биографии/события, как на скриншоте с норвежскими писателями).
- Кнопки **«Загрузить документ»** и **«Загрузить готовый индекс»** (важно: грузим не только небольшие доки, но и заранее посчитанные индексы — для тяжёлых корпусов).
- Выбор корпуса задаёт глобальный контекст для разделов Граф / Чат / Дашборд.

### 1.3 Раздел «Граф» (Explorer) — главный визуальный вау

- Интерактивный force-граф в стиле Obsidian: zoom / pan, перетаскивание узлов, подсветка соседей при наведении, цвет узла = тип сущности NEREL (29 типов), толщина ребра = strength связи.
- **Карточка сущности** (правая панель) при клике: ID, name, тип, LLM-описание, входящие/исходящие связи, исходные чанки (provenance).
- **Слой сообществ** (Leiden): переключатель «показать сообщества» — кластеры подкрашиваются, доступны их LLM-отчёты (title / summary / findings).
- Фильтры: по типам сущностей (как в текущем прототипе — легенда Entity Types), по сообществам, по строковому поиску узла; порог по strength связей.
- Контролы производительности для больших графов: лимит видимых узлов, кластеризация, «развернуть окрестность узла» по клику (важно — на скриншоте 2 граф из тысяч узлов превращается в «шар», нужна прогрессивная подгрузка).
- Связка с чатом: «спросить про эту сущность/сообщество» открывает чат с подставленным контекстом.

### 1.4 Раздел «Чат» (Agent)

- Диалог с агентом по активному графу. Выбор поискового движка RAGU: **LocalSearch / GlobalSearch / NaiveSearch / MixSearch / QueryPlanEngine** (из настроек или быстрым селектором в шапке чата).
- **Трейс ответа** (раскрываемая панель под сообщением) — ключевая фишка для демонстрации «почему RAGU, а не чёрный ящик»:
  - какие сущности / связи / чанки / сообщества подняты в контекст,
  - подсветка этих узлов на графе,
  - какой движок и пресет использованы, top-k, rerank вкл/выкл,
  - тайминги (retrieval / generation) и оценка энергии/стоимости запроса.
- Примеры-затравки вопросов под каждый датасет (чтобы пользователь не смотрел на пустой инпут).
- Стриминг ответа токенами.

### 1.5 Раздел «Настройки движка»

Реализуем **«средний» уровень**: пресеты + ручной режим. Slide-over панель, чтобы не загромождать главный экран; пара ключевых тумблеров вынесена в шапку.

**Пресеты (быстрый выбор в шапке):**
| Пресет | Идея | Профиль конфигурации |
|---|---|---|
| ⚡ **Быстро** | минимум шагов, низкая латентность | SimpleChunker · single-stage extraction · без ICL · без сообществ · NaiveSearch/LocalSearch · NanoVDB |
| 🎯 **Точно** | максимум качества | SmartSemanticChunker · two-stage + валидация + ICL(hybrid) · суммаризация · Leiden-сообщества · MixSearch + rerank + hybrid · Qdrant |
| 🎬 **Демо** | сбалансированно и наглядно показывает пайплайн | two-stage · сообщества вкл · LocalSearch + rerank · meno-lite 7B |

**Ручной режим — маппинг тумблеров на реальный движок RAGU** (это и есть «много ползунков и галочек», но осмысленно сгруппированных по 6 шагам пайплайна):

1. **Chunking:** стратегия (Simple / SemanticText / SmartSemantic), размер чанка, overlap.
2. **Extraction:** single-stage ↔ two-stage; валидация по NEREL вкл/выкл; ICL вкл/выкл + стратегия примеров (semantic / BM25 / hybrid / random); модель экстракции (**meno-lite 7B** / Qwen2.5-3B/7B/14B/32B).
3. **Summarization:** DBSCAN-дедупликация (eps, min_samples), LLM-суммаризация сущностей/связей вкл/выкл.
4. **Communities:** Hierarchical Leiden (resolution, max levels), суммаризация сообществ вкл/выкл.
5. **Refinement:** RemoveIsolatedNodes и прочие пост-обработчики.
6. **Search:** движок (Local/Global/Naive/Mix/QueryPlan), cross-encoder rerank вкл/выкл, hybrid dense+sparse (BM25 / BM42 / SPLADE) вкл/выкл, top-k; модель-генератор ответа; embeddings (bge-large-en / gte-multilingual).

> Не все настройки имеют смысл «на лету»: меняющие **индексацию** (шаги 1–5) применимы только при live-индексации/переиндексации; меняющие **поиск** (шаг 6) применяются к запросу мгновенно. В UI это разделяем визуально (бейдж «требует переиндексации»).

### 1.6 Дашборд

Четыре блока (по итогам уточнения — все включаем):
1. **Статистика графа:** узлы / связи / сообщества, распределение по типам сущностей NEREL, топ-сущности по степени, плотность, число изолированных узлов.
2. **Метрики качества:** из бенчмарков статьи — Answer Correctness, Evidence Recall, Context Relevancy, ROUGE-L; сравнение RAGU vs LightRAG / HippoRAG 2 / Naive (данные из таблиц 1–3).
3. **Стоимость и энергия:** оценка **кВт·ч по формуле `TDP × время × PUE`**, перевод в $ и в «экономию против GPT-4-class API» (база: ~40k токенов индексации/документ по тарифам gpt-4o из статьи). Бизнес-метрика «сколько вы сэкономили, крутя 7B локально».
4. **Тайминги пайплайна по шагам:** chunking → extraction → summarization → communities → refinement, плюс retrieval/generation для запросов; + **логи запросов и трейсы** (таблица последних запросов с разворачиваемым трейсом).

### 1.7 Экспорт и логи (сквозная функция)

Везде, где уместно, кнопка выгрузки:
- **Граф:** GraphML / GEXF / JSON (node-link) / PNG-снимок.
- **Сообщества:** отчёты Leiden (Markdown / JSON).
- **Чат:** транскрипт + трейсы (Markdown / JSON).
- **Дашборд:** метрики и тайминги (CSV / JSON).
- **Логи сервиса:** структурированные логи прогона (JSON Lines) для скачивания.

### 1.8 Что осознанно НЕ делаем в MVP

Аутентификация и личные кабинеты; платежи; мультитенантность; обучение/дообучение моделей из UI; редактирование графа руками; полноценный прод-кластер. Всё это — в бэклоге (глава 2).

## Глава 2. Идеи и бэклог (развитие)

### 2.1 Pre-MVP «nice to have» (если останется время до 10 июля)
- **Side-by-side сравнение конфигов:** один вопрос → два движка/пресета рядом (наглядно для видео: «LightRAG-режим vs RAGU-режим»).
- **Подсветка provenance прямо в исходном тексте** документа (какие предложения породили сущность).
- **Тур по интерфейсу** (онбординг-коачмарки) — полезно ревьюеру, который видит систему впервые.
- Готовые **«сценарии для видео»** — заранее заготовленные вопросы/действия, разворачивающие сильные стороны.

### 2.2 MVP-этап, ближе к концу
- **Live-индексация своего документа** на GPU с жёсткими лимитами (размер файла, длина очереди, демо-квота на сессию). Сначала mock прогресса по шагам, затем реальный backend.
- **Загрузка готового индекса** (NetworkX GML + векторное хранилище) — для тяжёлых корпусов без пересчёта.
- Реальные тайминги/энергия из бэкенда вместо оценок.

### 2.3 Post-MVP / Backlog (после подачи)
- **Деплой на Yandex Cloud:** Compute + GPU для инференса, объектное хранилище для индексов, managed Redis для очереди.
- **Прод-хранилища:** переключение NetworkX → **Neo4j**, NanoVDB → **Qdrant** (hybrid dense+sparse) — в коде RAGU это смена двух аргументов конструктора, в демо — переключатель «масштабный режим».
- **Аутентификация + квоты** на пользователя, шеринг ссылки на конкретный граф/запрос.
- **Публичный REST/Playground API** поверх RAGU (async-first пакет уже есть) для интеграторов.
- **Встраиваемый виджет графа/чата** (iframe / web-component) для сайтов клиентов.
- **Расширение корпусов** и пользовательские «коллекции», многоязычный контент.
- **A/B и оффлайн-оценка прямо в UI** (прогон бенчмарка на своём корпусе → метрики).
- **Версионирование графов** (инкрементальный upsert/update/delete у RAGU уже поддержан) — история изменений корпуса.
- **Наблюдаемость:** трейсинг (OpenTelemetry), метрики Prometheus/Grafana, алерты.

## Глава 3. Технологический стек

### 3.1 Общая архитектура (MVP)

```
┌──────────────┐   HTTPS/JSON + SSE    ┌─────────────────────────┐
│  Frontend    │ ───────────────────▶  │  API Gateway (FastAPI)  │
│  (SPA)       │ ◀─────────────────── │  + Pydantic-схемы        │
│  $mol/React  │   стриминг ответа     └───────────┬─────────────┘
└──────────────┘                                   │
                                ┌──────────────────┼───────────────────┐
                                ▼                  ▼                   ▼
                        ┌──────────────┐   ┌────────────────┐  ┌───────────────┐
                        │ RAGU engine  │   │ Index Store     │  │ Job Queue     │
                        │ (Python pkg) │   │ предындексы:    │  │ Redis + RQ/    │
                        │ async-first  │   │ GML + vector в  │  │ Celery (live  │
                        │              │   │ памяти/на диске │  │ индексация)   │
                        └──────┬───────┘   └────────────────┘  └──────┬────────┘
                               │                                       │
                               ▼                                       ▼
                        ┌──────────────┐                       ┌───────────────┐
                        │ LLM serving  │                       │ GPU worker     │
                        │ vLLM:        │                       │ (live index)   │
                        │ meno-lite 7B │                       │ Yandex Cloud / │
                        │ (mock в нач.)│                       │ локалка        │
                        └──────────────┘                       └───────────────┘
```

**Принцип «mock-first»:** фронт и API-контракт проектируем сразу с типизированными ответами; на старте бэкенд отдаёт зафиксированные предындексы и **замоканный** инференс/индексацию. По мере получения доступа к GPU (Yandex Cloud) подменяем mock-адаптеры на реальные — фронт не меняется.

### 3.2 Backend
- **Язык/раннер:** Python + **FastAPI** (RAGU — production-ready async-first Python-пакет, оборачиваем его API; синхронные обёртки уже автогенерируются).
- **Контракты:** Pydantic v2 (структурированные ответы движка уже валидируются Pydantic — переиспользуем типы).
- **Стриминг чата:** SSE / WebSocket.
- **Очередь live-индексации:** Redis + RQ (или Celery) — отдельный GPU-воркер, ограничения по размеру/квоте.
- **Хранилища (MVP):** предындексы как файлы (NetworkX **GML** + NanoVDB JSON) грузятся в память сервиса; KV — JSON. **Прод (бэклог):** Neo4j + Qdrant (смена двух аргументов конструктора BaseStorage).
- **Инференс:** vLLM с meno-lite 7B (одна consumer-GPU достаточно по статье). На старте — mock-LLM-сервер (он же используется в ~335 тестах RAGU — удобно для CI).

### 3.3 Граф-визуализация (ключевой технический выбор фронта)
- **Кандидаты:** Sigma.js (WebGL, тянет десятки тысяч узлов — важно, см. «шар» на скриншоте 2), Cytoscape.js (богатый API, layout-плагины), D3-force (гибко, но дороже на больших графах).
- **Рекомендация:** **Sigma.js + graphology** для основного рендера (производительность на больших корпусах) с серверной агрегацией/семплингом и прогрессивной подгрузкой окрестностей. Layout считаем на бэкенде (ForceAtlas2) и кешируем в предындексе, чтобы не пересчитывать в браузере.

### 3.4 Frontend stack — выбор и риски ($mol)

Контекст: фронтендеров в штате нет (ML/research/backend), но есть внешний знакомый, который сделал PR #58 на **$mol** и готов помогать. Запрос — остаться на $mol «ради эксперимента», если риски невелики; иначе уйти на мейнстрим.

**Что такое $mol.** Нишевый реактивный TypeScript-фреймворк (hyoo-ru/mam_mol, автор Дмитрий Карловский). Сильные стороны: точечные обновления реального DOM без virtual-DOM, ленивый рендер (не рендерит то, что вне видимой области — реально быстрый на больших списках), компактный рантайм, своя система компонентов и сборщик (`mam`), декларативный DSL `*.view.tree`.

#### Аналитика рисков $mol как основы

| Риск | Severity | Пояснение | Митигация |
|---|---|---|---|
| **Bus factor / зависимость от одного человека** | 🔴 Высокий | И фреймворк (фактически один мейнтейнер), и ваш носитель экспертизы — внешний человек вне команды. Уйдёт — поддерживать некому. | Зафиксировать его участие письменно; держать фронт **тонким** (вся логика в API), чтобы переписать UI было дёшево. |
| **Невозможность найма** | 🔴 Высокий | На рынке $mol-разработчиков практически нет; онбординг нового человека — недели. | Не критично, пока команда не растит фронт; но это блокер для «на будущее»-цели. |
| **Идиосинкразия и кривая входа** | 🟠 Средний | `$`-нейминг, `.view.tree` DSL, BEM, собственный бандлер — не переносятся на другие проекты, экзотичны для ревью. | Ограничить демо стандартными виджетами $mol; не изобретать сложных кастомных. |
| **Слабая поддержка со стороны AI-ассистентов** | 🟠 Средний | Мало кода/доков в обучающих данных → Claude Code/Copilot помогают по $mol заметно хуже, чем по React. Для вашего темпа (2 недели, AI-assisted) это прямой тормоз. | Для $mol-частей опираться на знакомого, а не на AI. |
| **Документация преимущественно на русском / разрозненная** | 🟡 Низкий | Затрудняет онбординг и не идеальна для международного контекста. | — |
| **Интеграция сторонних JS-библиотек (Sigma/Cytoscape, чарты)** | 🟠 Средний | Граф-виз и графики придётся оборачивать вручную в $mol-компоненты. | Изолировать их в «островках»; продумать обёртки заранее. |
| **Экосистема UI-китов / готовых компонентов** | 🟡 Низкий | Нет богатого набора готовых компонентов уровня MUI/shadcn → больше ручной верстки. | Достаточно для лаконичного демо. |

**Сильные стороны (для честности):** реальная производительность на больших реактивных вью (уместно для графа), компактный бандл, ленивый рендер, заинтересованный носитель экспертизы рядом.

#### Рекомендация
**Условно «за эксперимент», но с предохранителем.** $mol уместен, если выполняются три условия:
1. Внешний знакомый формально закреплён за фронтом на горизонт MVP + видео.
2. Фронт держим **тонким клиентом**: весь смысл (конфиги, трейсы, метрики) приходит с бэкенда типизированным; UI почти не содержит бизнес-логики.
3. **Граф-визуализация и чарты — в изолированных «островках»** на нативных JS-библиотеках, обёрнутых тонким $mol-адаптером.

Если хотя бы одно условие шатко (особенно №1) — **дефолт: React + TypeScript + Vite** (мейнстрим, лучшая AI-поддержка под ваш сжатый срок, нанимаемость, готовые UI/граф-библиотеки). 

> Практический компромисс под дедлайн: **API-контракт и mock-данные — фреймворк-агностичны** (просто JSON-схемы). Тогда выбор $mol vs React не блокирует старт: каркас (этот wireframe) и контракт делаем сразу, а финальную реализацию кладём на тот стек, носитель которого реально доступен. Риск «застрять на $mol без помощи» этим снимается.

### 3.5 Деплой
- **MVP / для видео:** Docker Compose (фронт + FastAPI + Redis + mock-LLM) локально или на одной машине с GPU + публичный туннель (Cloudflare Tunnel / ngrok) для ссылки ревьюерам.
- **Будущее:** Yandex Cloud — Compute Instance с GPU под vLLM, Object Storage под индексы, Managed Redis под очередь, Container Registry, балансировщик. Тот же Docker Compose → managed-сервисы.

### 3.6 Тесты и CI/CD (детали — в роадмапе)
- Бэкенд: pytest + детерминированный mock-LLM-сервер (как в RAGU, ~335 тестов — прогон за секунды, без API-ключей).
- Контрактные тесты API (схемы Pydantic ↔ фикстуры фронта).
- Фронт: smoke/e2e (Playwright) на ключевых сценариях.
- CI: GitHub Actions (lint + tests на каждый PR); CD — деплой по тегу. Накатываем после стабилизации каркаса.
